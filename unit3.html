Logical DFD vs Physical DFD

Aspect Logical DFD Physical DFD
Focus Highlights what the system does functionally. Explains how the system works operationally.
Processes Represents business tasks or functions. Shows specific programs, modules, or manual tasks.
Data Stores General collections of data without specifying storage. Includes physical files, databases, or manual records.
External Entities Logical roles, systems, or users interacting with the system. Devices, departments, or specific entities like staff roles.
Implementation Independent of hardware and software details. Dependent on technology and resources.
Purpose To understand business processes and user requirements. To provide a detailed blueprint for system implementation.

Steps of Prototype Method

  1. Requirement Gathering:
    • Collect user expectations and functional requirements.
    • Define the scope and prioritize features for the prototype.
  2. Quick Design:
    • Draft a basic structure of the system.
    • Focus on interface layouts, workflows, and key processes.
  3. Build Prototype:
    • Create a working model to demonstrate functionality.
    • Include primary features with minimal backend processing.
  4. User Evaluation:
    • Share the prototype with users for feedback.
    • Identify strengths, weaknesses, and gaps in functionality.
  5. Refine Prototype:
    • Enhance the prototype using user suggestions.
    • Repeat iterative improvements until it meets expectations.
  6. Implement and Maintain:
    • Finalize the system for production deployment.
    • Ensure ongoing updates to maintain reliability and usability.
