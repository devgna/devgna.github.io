Unit 1: Introduction to .NET Framework

1. Define CLR (Common Language Runtime) and explain its role.

The CLR (Common Language Runtime) is the core execution engine of the .NET Framework. It's an environment that manages the execution of .NET code (managed code).

Key Roles:

  • MSIL to Native Code: It converts the intermediate code (MSIL or CIL) into native machine code using a Just-In-Time (JIT) compiler.
  • Memory Management: It automatically handles memory, most importantly through Garbage Collection (GC), which reclaims memory from objects that are no longer in use.
  • Security: It provides services like Code Access Security (CAS) to prevent unauthorized access.
  • Exception Handling: It provides a standard way to handle errors (exceptions) across different .NET languages.
  • Type Safety: It enforces the Common Type System (CTS) to ensure that different .NET languages can interact safely.

The .NET Framework architecture is a layered model that provides services for building, deploying, and running applications.

Main Components:

  1. Common Language Runtime (CLR): The "engine" at the bottom that executes code (as described in Q1). It includes the JIT compiler, Garbage Collector (GC), and handles security and exception handling.
  2. Framework Class Library (FCL): A huge library of pre-written, reusable code (classes, interfaces, value types) that developers can use for common tasks like database access (ADO.NET), file I/O, string manipulation, etc.
  3. Common Language Specification (CLS): A set of rules that language compilers must follow to create code that can interoperate (work with) other .NET languages.
  4. Common Type System (CTS): Defines a set of common data types that all .NET languages share, ensuring (for example) that a C# int is the same as a VB.NET Integer.
  5. Application Layers: These are the types of applications you can build, which all use the FCL and CLR (e.g., Windows Forms, ASP.NET, WPF).

4. What is the Common Type System (CTS) and how does it support language interoperability?

The Common Type System (CTS) is a standard set of data types defined by the .NET Framework that all .NET languages must use.

How it supports interoperability:

  • It ensures that all .NET languages "speak the same language" regarding data.
  • For example, a C# int and a VB.NET Integer both compile down to the same type in the CLR: System.Int32.
  • Because all languages map their types to this common system, it allows a class written in C# to be seamlessly used (or inherited) by a class written in VB.NET, and vice versa.

5. Explain Garbage Collection (GC) in .NET.

Garbage Collection (GC) is the automatic memory manager in the CLR. Its job is to find and release memory that is no longer being used by the application.

How it works (simplified):

  1. When an object is created, it's placed in a memory area called the "managed heap."
  2. The GC periodically runs and checks all objects in the heap.
  3. It identifies which objects are "unreachable" (i.e., no part of the running program still holds a reference to them).
  4. It releases the memory occupied by these unreachable objects.
  5. It compacts the remaining objects in memory to avoid fragmentation.

↑

The main benefit is that it prevents common programming errors like memory leaks, where memory is allocated but never freed.

Unit 2: C# Basics & OOP

1. List and describe the four main principles of OOP.

  1. Encapsulation: The practice of binding data (fields) and the methods that operate on that data into a single unit (a class). Data is often hidden (made private) and can only be accessed via public methods (getters/setters or properties).
  2. Inheritance: A mechanism where a new class (child/derived class) can "inherit" the properties and methods of an existing class (parent/base class). This promotes code reusability.
  3. Polymorphism: This means "many forms." It allows objects of different classes to be treated as objects of a common base class. It's often implemented using method overriding (where a child class provides its own version of a parent's method) or method overloading (having multiple methods with the same name but different parameters).
  4. Abstraction: Hiding complex implementation details from the user and only showing the essential features. This is achieved using abstract classes and interfaces, which define *what* a class should do, but not *how* it should do it.

3. What are Boxing and Unboxing in C#? Provide examples.

Boxing and Unboxing are processes that bridge the gap between Value Types and Reference Types.

  • Boxing (Implicit): The process of converting a Value Type (like int) into a Reference Type (like object). The CLR wraps the value inside a new object on the heap.
  • Unboxing (Explicit): The process of extracting the original Value Type from the object it was boxed into. This requires an explicit cast.

// --- Unboxing ---
int myUnboxedValue = (int)myBoxedValue; // Explicitly unboxed.

4. Explain Checked and Unchecked blocks for overflow handling.

These blocks tell the CLR how to handle "arithmetic overflow," which happens when the result of a math operation is too large for the data type to hold.

  • checked block: This block *enforces* overflow checking. If an overflow occurs, a System.OverflowException is thrown. This is safer.
  • unchecked block: This block *ignores* overflow. The value will simply "wrap around" (becoming a negative number, usually) and no exception is thrown. This is the default behavior in C# for most operations.

5. (CODE) Create a C# class using auto-implemented properties.

Auto-implemented properties are a shortcut to create a private backing field and its public get and set accessors in one line.


using System;

// 1. Define the class with auto-implemented properties
public class Book
{
    // Auto-property for Title. Compiler creates a private 'title' field.
    public string Title { get; set; }

    // Auto-property for Author. Compiler creates a private 'author' field.
    public string Author { get; set; }

    // Auto-property for Price.
    public double Price { get; set; }

    // Constructor to set values
    public Book(string title, string author, double price)
    {
        Title = title;
        Author = author;
        Price = price;
    }

    // Method to display details
    public void DisplayDetails()
    {
        Console.WriteLine($"Title: {Title}");
        Console.WriteLine($"Author: {Author}");
        Console.WriteLine($"Price: {Price:C}"); // 'C' formats as currency
    }
}

// 2. Use the class
public class Program
{
    public static void Main(string[] args)
    {
        // Create an instance
        Book myBook = new Book("Fundamentals of Visual Programming", "Dr. Patel", 49.99);

        // Use the 'get' accessors via the method
        myBook.DisplayDetails();
        
        // Use the 'set' accessor
        myBook.Price = 54.99;
        Console.WriteLine("\nPrice updated:");
        myBook.DisplayDetails();
    }
}

TextBox Control: Allows the user to enter text.

    • Click: The most important event. Fires *once* when the user clicks the button.

3. What is the role of the ErrorProvider control?

The ErrorProvider is a control used for form validation. It does *not* stop the user like a MessageBox. Instead, it provides a non-intrusive way to show that a control has an error.

How it works:

  1. You add an ErrorProvider control to your form (it sits in the component tray).
  2. In your code (e.g., a Button's Click event), you check if a control is valid.
  3. If the control is *invalid* (e.g., a TextBox is empty), you call the ErrorProvider's SetError() method.
  4. This displays a small, blinking error icon (e.g., an "!") next to the invalid control.
  5. When the user hovers over the icon, the error message you provided is shown as a tooltip.

1. Explain the role of HTTP Request and HTTP Response.

HTTP (HyperText Transfer Protocol) is the foundation of data communication for the World Wide Web. It works as a request-response protocol between a client (like a web browser) and a server.

  • HTTP Request: This is the message sent by the client (e.g., your browser) to the server when you want to access a resource (like a webpage or an image). It includes:
    • A request line (e.g., GET /index.html).
    • Headers (e.g., information about your browser, what data types it accepts).
    • An optional body (used to send data, like in a POST request).
  • HTTP Response: This is the message sent back by the server to the client. It includes:
    • A status line (e.g., 200 OK or 404 Not Found).
    • Headers (e.g., information about the content, like Content-Type: text/html).
    • The actual resource (the HTML, image, or data) in the response body.

3. Describe the main stages of the ASP.NET Page Life Cycle.

The ASP.NET Page Life Cycle is a series of steps a page goes through from the time it's requested by the server to the time it's rendered in the browser. Understanding this is key to knowing *when* to write your code.

  1. PreInit: The very first step. Good for setting a Master Page or Theme dynamically.
  2. Init: All controls on the page are initialized, and their UniqueID is set.
  3. InitComplete: All initialization is complete. ViewState tracking is turned on.
  4. OnPreLoad: ViewState and PostBack data are loaded into the controls.
  5. Load: The main event for page logic. The page and all controls are fully loaded. This is where you typically check if (!IsPostBack) to set up data on the first load (e.g., database queries).
  6. Control PostBack Event(s): If the page was submitted (PostBack), the specific control event that caused it (e.g., a Button_Click) runs now.
  7. LoadComplete: All processing and event handling is finished.
  8. OnPreRender: The last chance to make changes to the page or its controls before they are rendered into HTML.
  9. OnSaveStateComplete: ViewState and ControlState are saved. No further changes are possible.
  10. Render: The page and its controls are "drawn" as HTML, DHTML, and script to be sent to the browser.
  11. Unload: The page is removed from memory. Used for final cleanup (e.g., closing database connections or files).

4. What are Validation Controls in ASP.NET?

Validation Controls are server controls that automatically check user input on a form to ensure it's valid. They can run on both the client (using JavaScript) and the server, preventing bad data from being submitted.

  • RequiredFieldValidator: Ensures the user does not leave a control (like a TextBox) empty.
  • RangeValidator: Checks that the user's input is within a specified range (e.g., age between 18 and 65).
  • CompareValidator: Compares the user's input against a fixed value or the value of another control (e.g., "Password" and "Confirm Password" fields must match).
  • RegularExpressionValidator: Checks that the input matches a specific pattern (a "regex"), such as for a valid email address, phone number, or postal code.
  • CustomValidator: Allows you to write your own custom logic (in C# or JavaScript) for complex validation rules.
  • ValidationSummary: Gathers all the error messages from the other validators on the page and displays them in a single, collective list.

5. What is the role of the web.config file? How does it differ from machine.config?

Both are XML files used for configuration in .NET applications, but they have different scopes.

  • machine.config:
    • This is the "master" configuration file for the *entire machine* (server).
    • It defines the default settings for *all* .NET applications running on that server.
    • It is located in the main .NET Framework directory.
    • Settings here are inherited by all applications, but can be overridden.
  • web.config:
    • This is the configuration file for a *specific* web application.
    • It is placed in the root directory of your application.
    • It is used to *override* the default settings from machine.config and define application-specific settings.
    • Common settings include: database connection strings, session state settings, security rules, and custom error pages.

2. Analyze the role of DataSet vs. DataReader.

DataReader (Connected):

  • Analogy: A fast-moving, forward-only "data firehose."
  • It's a lightweight component that reads a stream of data directly from the database.
  • It is forward-only (you can only read records one after another, you can't go back).
  • It is read-only (you cannot use it to change data).
  • It is extremely fast and uses very little memory, as it only holds one record in memory at a time.
  • Requires the SqlConnection to remain open the entire time you are reading.

DataSet (Disconnected):

  • Analogy: An in-memory, mini-database.
  • It's a heavy-duty object that holds a *full copy* of data in your application's memory.
  • It is fully scrollable and editable. You can navigate it forwards, backwards, and make changes.
  • It can hold multiple DataTable objects and their relationships (DataRelation).
  • It is disconnected. A SqlDataAdapter is used to "Fill" the DataSet from the DB. After the fill, the database connection is closed. All work (reading, editing) is done on the local copy. The DataAdapter is used again later to "Update" the database with any changes.

3. Explain the main ADO.NET components (Connection, Command, DataAdapter).

  1. Connection (SqlConnection):
    • Represents the actual connection to a specific database.
    • It holds the Connection String, which contains the address (Data Source), database name (Initial Catalog), and security info (User ID, Password, or Integrated Security).
    • Its main methods are Open() and Close().
  2. Command (SqlCommand):
    • Represents a single SQL statement (like SELECT, INSERT, UPDATE, DELETE) or a stored procedure.
    • It *must* be associated with a SqlConnection object to know *where* to execute the query.
    • It has the `Execute...()` methods (see next question).
  3. DataAdapter (SqlDataAdapter):
    • This is the "bridge" for the disconnected model.
    • It holds a SelectCommand (and optionally Insert, Update, Delete commands).
    • Its Fill() method opens the connection, runs the SelectCommand, loads all the results into a DataSet, and then closes the connection.
    • Its Update() method connects to the DB and pushes any changes from the DataSet back to the database.

4. Explain the ExecuteReader(), ExecuteScalar(), and ExecuteNonQuery() methods.

These are the three main methods of the SqlCommand object:

  • ExecuteReader():
    • Use: For SELECT statements that return *multiple rows*.
    • Returns: A SqlDataReader object.
    • Model: Used in the Connected architecture. You must loop through the reader (e.g., while (myReader.Read()) { ... }) to get the data, and the connection must stay open.
  • ExecuteScalar():
    • Use: For SELECT queries that return *a single value*.
    • Returns: An object. It returns only the value from the first column of the first row.
    • Example: SELECT COUNT(*) FROM Users or SELECT MAX(Price) FROM Products.
  • ExecuteNonQuery():
    • Use: For SQL commands that do *not* return rows of data.
    • Returns: An int representing the number of rows affected by the command.
    • Example: INSERT, UPDATE, and DELETE statements.

5. (CODE) Write C# code to fill a DataSet and bind it to a DataGridView.

using System;
using System.Data;
using System.Data.SqlClient;
using System.Windows.Forms;

// ... inside your Form class ...

private void LoadDataButton_Click(object sender, EventArgs e)
{
    // 1. Connection string
    string connString = @"Data Source=YourServer;Initial Catalog=YourDatabase;Integrated Security=True";

    // 2. The query to run
    string query = "SELECT * FROM Employees";

    // 3. Create the DataAdapter and DataSet
    // The DataAdapter needs the query and connection string
    SqlDataAdapter dataAdapter = new SqlDataAdapter(query, connString);

    // The DataSet is the in-memory copy of the data
    DataSet dataSet = new DataSet();

    try
    {
        // 4. Fill the DataSet
        // The Fill method automatically opens the connection,
        // runs the query, loads the data into 'dataSet',
        // and closes the connection.
        dataAdapter.Fill(dataSet, "EmployeesTable"); // "EmployeesTable" is a friendly name

        // 5. Bind the DataSet to the DataGridView
        // We bind to the specific table inside the DataSet
        dataGridView1.DataSource = dataSet.Tables["EmployeesTable"];
    }
    catch (Exception ex)
    {
        MessageBox.Show("Error loading data: " + ex.Message);
    }
    finally
    {
        // Good practice to dispose of the adapter
        if (dataAdapter != null)
        {
            dataAdapter.Dispose();
        }
    }
}
