Unit 4: System Design

1) Introduction to Systems Design

Systems design is the process of translating user requirements into a structured format that can be implemented in software. During this phase, system analysts take the abstract concepts from the System Requirements Specification (SRS) document and develop a detailed design that will guide coding and implementation. This phase bridges the gap between understanding the problem and defining the solution.

2) System Design Levels

  1. Architectural Design:
    • The highest-level design phase, focusing on the overall structure of the system.
    • Identifies the major components and their interactions.
    • Abstract representation of the system to give designers an idea of the proposed solution.
    • Aims to ensure that the system’s architecture will support scalability, flexibility, and future enhancements.
  2. High-Level Design:
    • Breaks the architectural design into subsystems and modules.
    • Focuses on how the system’s components can be implemented and how they will interact.
    • Identifies the modular structure, ensuring each part of the system can operate independently while working together as a whole.
  3. Detailed Design:
    • Deals with the actual implementation of modules defined in the high-level design.
    • Focuses on logical structures for each module and how they interact with other modules.
    • Specifies data flows, user interfaces, and any specific implementation details needed for coding and testing.

3) Elements of Design

  1. Data Flows: Movement of data into, out of, and within the system.
  2. Data Stores: Temporary or permanent storage of data used by the system.
  3. Processes: Activities that receive, manipulate, and deliver data or information. These can be manual or computer-based.
  4. Procedures: Methods or routines for achieving desired outcomes within the system.
  5. Controls: Standards and guidelines that ensure activities occur as intended and provide mechanisms for correcting problems if detected.
  6. Roles: Responsibilities of all persons involved with the system, from data entry to the distribution of outputs or results.

4) Design of Output System

Output refers to the results or information generated by the system, which are presented to the user. When designing an output system, the system analyst must consider:

  1. What Information to Present:
    • Determine the essential data and results that need to be displayed or printed.
  2. Output Medium:
    • Decide whether the output will be displayed on a screen, printed on paper, or spoken by the system.
  3. Presentation Format:
    • Arrange the information in a format that is clear and usable for the recipient (e.g., tabular, graphical, etc.).
  4. Distribution Method:
    • Plan how the output will be distributed to the intended recipients.
  5. Layout:
    • The arrangement of information on displays or printed documents, including headings, columns, pagination, and data fields.

Output Objectives: - Convey past activities, current status, or future projections. - Signal important events, such as errors or opportunities. - Trigger actions or confirm actions made by the user.

Types of Output: - Reports: Detailed summaries or breakdowns of system activities. - Messages: Notifications of important system events or actions. - Documents: Help manuals or transaction receipts.

Key Output Questions: - Who will receive the output? - What is the output’s purpose? - How much detail is needed? - When and how often is the output needed? - Method of presentation (e.g., printed, displayed on screen).

5) Design of Input System

Input design focuses on data entry into the system, ensuring that only necessary and valid data is captured. The system analyst must decide:

  1. What Data to Input:
    • Identify the data needed for processing transactions, distinguishing between variable (user-input) and constant (system-generated) data.
  2. Input Medium:
    • Choose methods for capturing data, such as keyboard entry, scanners, or punch cards.
  3. Data Arrangement and Coding:
    • Specify how the data should be arranged and coded to facilitate processing.
  4. Data Validation:
    • Design mechanisms to validate data during entry to prevent errors.

Objectives of Input Design: - Controlling input volume: Ensuring only necessary data is entered. - Avoiding delays: Reducing bottlenecks during data entry. - Avoiding errors: Ensuring data integrity and correctness. - Simplifying the process: Making input easy and intuitive.

6) Form Design

Forms are the interface between the user and the system. They are used to input data into the system. Good form design is crucial to make the data entry process effective and efficient.

  1. Layout:
    • Organize fields in a logical order. Group related items together to make the form easy to understand and complete.
  2. Captions:
    • Provide clear instructions or labels to help users understand what data is required.
  3. Data Capture Methods:
    • Determine how data will be captured (e.g., keyboards, scanners, etc.).

Good Form Design Principles: - Ease of Completion: Forms should be intuitive and easy to fill out. - Accuracy: Forms should ensure that all required data is entered correctly. - Attractiveness: Forms should be visually appealing, with clean layouts and clear instructions. - Logical Flow: Organize form sections such as Heading, Identification, Instructions, Body, and Comments.

7) Design of Control

The system analyst must design controls to ensure data integrity and security. Input controls are used to validate that the data entered is correct, authentic, and complete.

Types of Controls: - Input Controls: Ensure only authorized users can access the system and that transactions are acceptable. - Error Detection: Identify issues with input and specify how to handle errors. - Security and Authentication: Validate user credentials and prevent unauthorized access.

8) Design of User Interface

An interface is a boundary between the user and the computer system, allowing the user to interact with the system.

Objectives of the Interface Design: - Guide User Actions: Help users understand what actions are possible. - Minimize Errors: Reduce the chance of user mistakes by providing clear prompts and feedback.

Actions Occurring at the Interface: 1. Navigation: Users choose which information to display or how to move through the system. 2. Processing: Data entry, editing, storage, and retrieval. 3. Messages: Provide feedback or alerts to the user.

Guidelines for Interface Design: - Consistency: Ensure the interface behaves predictably across different screens. - Feedback: Provide informative feedback to the user after each action. - Error Prevention: Help prevent errors and provide simple error handling options. - Reversal of Actions: Allow users to undo their actions if needed.

9) Design of Database Interactions

Once the basic processes, inputs, and outputs are defined, the system analyst must decide on how the system’s data will be structured and stored. This involves defining the database schema, tables, and relationships within the system.

Objectives of Database Design: - Ensure efficient storage and retrieval of data. - Maintain data integrity and consistency. - Ensure availability of data to different modules. - Prevent data redundancy and anomalies.

Database Design Concepts: 1. Data Models: Define the structure of data (e.g., Relational Model, Entity-Relationship Model). 2. DBMS: A Database Management System like Oracle or MySQL helps manage the creation, updating, and querying of databases. 3. Database Schema: The structure that describes how data is organized, including tables, fields, and relationships. 4. Transactions: Define operations like data insertion, updates, and retrievals that need to be executed as atomic units.
