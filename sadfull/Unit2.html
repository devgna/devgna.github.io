Unit 2: Classical Systems Development Methodology

1) System Development Life Cycle (SDLC)

Definition: The System Development Life Cycle (SDLC) is a structured methodology used to develop information systems. It involves policies and procedures for developing or altering systems throughout their life cycle, aiming to create high-quality systems that meet customer expectations while being completed on time and within budget.

Phases of SDLC:

  1. Feasibility Study/Planning:
    • Define the problem and objectives.
    • Assess the scope of the existing system and determine the objectives of the new system.
    • Prepare a project schedule.
    • Identify threats, constraints, integration, and security concerns. A feasibility report is produced at the end of this phase.
  2. Analysis and Specification:
    • Gather, analyze, and validate system requirements.
    • Define the system requirements and create prototypes.
    • Prioritize requirements and analyze alternative solutions.
    • A Software Requirement Specification (SRS) document is prepared at the end of this phase.
  3. System Design:
    • Design the system’s application, network, databases, user interfaces, and system interfaces.
    • Transform the SRS document into a logical structure, which contains specifications for implementation.
    • Prepare a design document, which includes training, contingency plans, and maintenance plans.
  4. Implementation:
    • Convert the design into source code.
    • Integrate the system modules and test for errors.
    • Generate a test report, which includes error detection.
    • Deploy the system into the work environment.
  5. Testing:
    • Conduct unit testing, integration testing, system testing, and acceptance testing.
    • Ensure the system meets all requirements.
    • Resolve any residual errors after the testing phase.
  6. Deployment:
    • Deploy the software once it is verified.
    • If required, release the software with suggested enhancements.
    • Begin system maintenance after deployment.
  7. Maintenance/Support:
    • Offer phone or on-site support for users after system installation.
    • Implement changes or new requirements after software deployment.

2) Information Gathering Techniques

Purpose: To determine the information requirements of an organization and prepare a precise Software Requirement Specification (SRS) document.

Techniques:

  1. Interviewing:
    • Unstructured Interviews: Conducted with general topics in mind, allowing open-ended responses.
    • Structured Interviews: Predefined set of questions with close-ended answers.
    • Advantages: Good for qualitative information, helpful in discovering key problems, and reducing misunderstandings.
  2. Questionnaires:
    • Free-format Questionnaires: Respondents provide their answers freely.
    • Fixed-format Questionnaires: Predefined response choices (multiple-choice, ranking, etc.).
    • Advantages: Useful for gathering information from large groups, providing confidentiality, and collecting statistical data.
  3. Review of Records, Procedures, and Forms:
    • Review existing records and procedures to understand system capabilities and document current operations.
  4. Observation:
    • Observing the organization’s operations to understand the current system requirements and document the necessary changes.
    • Advantages: Provides accurate data and is useful when authenticity or complexity of data is a concern.
  5. Modern Methods:
    • Joint Application Development (JAD): Intensive workshops with users, managers, and analysts to define system requirements.
    • Prototyping: A working model of the system is created to collect user feedback and refine requirements.
    • Group Support Systems (GSS): Tools for collaborative decision-making and gathering system requirements.
    • CASE Tools: Tools used to support software engineering activities and ensure a structured approach to system development.

3) Software Requirements Specification (SRS)

Definition: An SRS is a document that describes the software system to be developed. It outlines the system’s functional and non-functional requirements and provides the basis for design, implementation, and testing. The SRS ensures that the development team understands the customer’s requirements.

Goals of an SRS:

  1. Feedback to the Customer: It assures the customer that the development team understands the issues to be addressed.
  2. Breaking the Requirements Down: Writing the requirements in a structured manner helps organize the information and break down the problem into smaller components.
  3. Facilitating Other Documentation: The SRS serves as the foundation for other documents like the software design specification.
  4. Product Validation: It serves as a reference for testing and validating the system.

SRS Format:

  1. General Description:
    • The product’s purpose, features, and benefits.
    • Description of the target user community.
  2. Functional Requirements:
    • Describes the expected behavior of the system, detailing the inputs, outputs, and the system’s operations (e.g., calculations, data processing).
  3. Interface Requirements:
    • Specifies how the software communicates with other systems or users, including communication protocols, data formats, etc.
  4. Performance Requirements:
    • Describes how the system should perform under certain conditions, such as speed, memory usage, or error rates.
  5. Non-Functional Attributes:
    • Includes attributes such as security, portability, reliability, and scalability.
  6. Preliminary Schedule and Budget:
    • Outlines the initial project timeline and budget required for development.
