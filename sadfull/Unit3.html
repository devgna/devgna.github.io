Unit 3: Introduction to Data flow diagrams and Prototype design

1) System Design and Data Flow Diagrams (DFDs)

System design is the process of transforming user requirements into a structured system that can be coded and implemented. The System Requirements Specification (SRS) document defines what the system needs to do, while the design phase translates these requirements into more specific terms for implementation. System design represents a shift from understanding the problem to focusing on the solution, outlining how to fulfill the system’s functional requirements.

2) Data Flow Diagrams (DFD)

Data Flow Diagrams (DFDs) are a graphical tool used to represent the flow of data within a system, detailing where it is processed and how it moves through the system. DFDs help communicate the system’s functionality to users, designers, and analysts.

Key Features of DFD: - Developed by Larry Constantine. - Expresses how data flows between processes and external entities. - Graphical representation of system requirements. - Requires iterations to refine for accuracy and completeness.

DFD Notations:

  1. External Entities: External actors or systems interacting with the system. Examples: Supplier, Customer.
  2. Processes: Activities or functions that transform data from input to output. Examples: Calculate Pay, Reserve Product.
  3. Data Flows: Movement of data between processes, data stores, and external entities. Examples: Customer Information, Order Details.
  4. Data Stores: Locations where data is stored. Examples: Student Records, Order History.

Levels of DFD:

  1. Context-Level DFD: The highest-level DFD, showing the entire system as a single process and interactions with external entities.
  2. Level 1 DFD: Breaks down the context-level diagram into major processes or sub-systems and shows how they interact.
  3. Level 2 DFD: Further decomposes the Level 1 processes into more detailed processes or sub-processes.

3) Logical DFD vs Physical DFD

  • Logical DFD: Focuses on what the system does (business processes and data flows). It is independent of hardware and technology and represents the logical flow of data within the system.
  • Physical DFD: Focuses on how the system is implemented, including hardware, software, and physical devices. It shows specific programs, devices, and people involved in the data flow and processes.
Aspect Logical DFD Physical DFD
Focus Describes business functions (what the system does) Describes system implementation (how it works)
Processes Business activities or tasks Programs, program modules, manual tasks
Data Stores Abstract representation of data storage Specific files, databases, or physical records
Entities External systems or roles Real-world devices, departments, people
Purpose Represents business functionality Represents the system’s actual operations

4) Prototype Design

A prototype is an early, working version of the system that is used to gather feedback from users. It is typically incomplete, focusing only on core features and intended to evolve into the final product. Prototyping helps clarify requirements and reduce uncertainty in system design.

Advantages of Prototyping:

  • User Involvement: Users can provide feedback early in the development process.
  • Cost-Effective: Problems are identified early, reducing development costs.
  • Faster Feedback: Prototypes allow quick iterations and refinement based on user input.
  • Reduced Risk of Missing Features: Helps ensure the final system has all necessary features.

Disadvantages of Prototyping:

  • Insufficient Requirement Analysis: If the prototype is too rudimentary, key requirements might be overlooked.
  • Misleading for Users: Users may mistake the prototype for the final system, leading to unrealistic expectations.
  • Scope Creep: The continuous addition of features during the prototyping phase may expand the system beyond its initial scope.

Steps in Prototyping:

  1. Requirement Gathering: Collect basic requirements from users, focusing on core needs.
  2. Quick Design: Create an initial design that outlines the system’s basic structure.
  3. Build Prototype: Develop the working model with limited features.
  4. User Evaluation: Present the prototype to users for feedback on functionality and usability.
  5. Refine Prototype: Make iterative improvements based on feedback.
  6. Implementation: Once the prototype is refined, proceed to the full implementation of the final system.

Types of Prototypes:

  • Throwaway Prototype: Built quickly, used for feedback, and then discarded after gathering information.
  • Evolutionary Prototype: Continually refined and evolved into the final system.

5) Tools for Prototyping

  1. Fourth Generation Languages (4GL): High-level programming languages that allow for quicker development. These tools focus more on the desired output than on the specifics of how tasks are performed (e.g., SQL for database queries, report generators).
  2. Application Generators: Tools that help create entire applications with high-level specifications. These tools can automatically generate the necessary code for common tasks.
  3. Screen Generators: Software that allows for the design of interactive user interfaces, including input validation and processing capabilities.
  4. Data Dictionary Systems: These systems store data definitions and descriptions for all data used within an organization, making it easier to manage and maintain data consistency across the system.
  5. Reusable Code Libraries: Collections of pre-written code that can be used across multiple applications, reducing the time spent writing common functionality.

6) Common Misconceptions About Prototyping

  1. Prototyping is only for simple applications: Prototypes can be used for both small and large, complex systems to reduce uncertainty and ensure user needs are met.
  2. It’s a trivial activity: Building and testing a prototype requires careful planning and a structured approach, even if the prototype is temporary.
  3. Prototypes are for small projects only: Prototyping is valuable for both small and large projects, particularly when user input and feedback are crucial to success.
  4. Token user involvement: Prototypes are designed to engage users actively, not just for show. They help users interact with the system and provide real, actionable feedback.
